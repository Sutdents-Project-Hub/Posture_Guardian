# 專案範圍、架構與實作順序

## 問題與目標使用者

核心使用者是長時間坐著讀書、容易在專注時失去姿勢覺察的中學生。真正要解決的不是「使用者不知道要坐正」，而是：

- 專注時無法察覺姿勢已偏離。
- 固定鬧鐘與當下姿勢無關，容易被關閉。
- 沒有可解釋指標與長期回饋，使用者看不到改善。

產品承諾是「當偏移真的持續時才提醒，讓使用者看得到自己逐步改善」，不宣稱矯正脊椎、診斷疾病或取代醫療專業。

`app/` 與 `backend/` 分別是 Expo 與 FastAPI 的固定 component root；manifest 直接位於 component 根目錄。此架構不增加 project-name、framework-name 或其他分類包層。

## 系統邊界

| 邊界 | 責任 | 不負責 |
|---|---|---|
| `app` | 相機權限、視角引導、校準、即時結果、提醒、工作階段與趨勢畫面 | 保存秘密、直接連 PostgreSQL、在前端呼叫具秘密的 AI API |
| `backend` | 接收暫存影格／節點、MediaPipe 推論、指標與規則、資料 API、AI provider | 永久保存原始影像、直接控制危險硬體 |
| PostgreSQL | 工作階段、摘要指標、提醒事件、使用者偏好與同意紀錄 | 原始影格、未經同意的聯絡人資料 |
| Microsoft Foundry | 根據去識別摘要產生建議，形成 Azure 概念驗證 | 決定姿勢是否良好、做醫療診斷 |
| Coolify | 部署 client、API 與 PostgreSQL | 取代 Azure 概念驗證或自動取得競賽所需額度 |

## 決賽 MVP 垂直切片

1. 手機或 Web 進入相機設定，選擇側面／正面視角。
2. 引導使用者保持自然中性坐姿 10 秒，建立個人基線。
3. 每秒約 2 幀取得姿態節點，低品質影格不參與判定。
4. 使用者刻意低頭／前傾／歪斜並持續 8 秒後，畫面顯示哪一項偏移以及如何調整。
5. 回正後停止提醒，完成工作階段並顯示良好坐姿比例、提醒次數與反應時間。
6. API 將摘要寫入 PostgreSQL；Microsoft Foundry 以摘要產生一段個人化建議。
7. Azure 不可用時顯示「規則式建議」，仍可完成主要流程且不冒充 Azure 結果。

## 可驗收條件

| 能力 | 決賽前完成條件 |
|---|---|
| 校準 | 正面／側面模式皆能拒絕缺少必要節點或人體未入鏡的校準 |
| 姿勢事件 | 只在有效影格達門檻且偏移持續 8 秒時建立事件 |
| 可解釋性 | 每次提醒至少顯示偏移項目、目前值、個人基線與門檻 |
| 介入階段 | 預設初期，可手動修改；具足夠工作階段後才允許自動升降級 |
| 資料 | 資料庫不含原始圖片；能查詢單次工作階段摘要 |
| AI | prompt 只含摘要；Foundry 成功與 fallback 都有明確來源標籤 |
| 展示 | 同一流程可用真人鏡頭與預錄測試片段各完成一次 |
| 說明 | 能清楚說明支援姿勢、未支援情境、測試樣本與目前實測結果 |

## 明確不納入決賽 MVP

- 腰帶按壓、馬達施力或其他可能造成不適的硬體。
- 醫療診斷、疼痛治療、脊椎曲率量測或「預防疾病」宣稱。
- 班級／企業監控、排行榜、付費訂閱與 App Store 正式上架。
- 儲存原始照片／影片、無同意的人臉辨識或將所有顧客資料直接送給生成式 AI。
- 在缺乏測試資料時宣稱準確率大於 90%、改善學習效率或降低就醫率。
